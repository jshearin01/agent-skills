# AI Slop Detector & Remediator

## Overview

AI slop is the statistical median of training data made visible — content that is technically correct, functionally passable, but instantly recognizable as machine-generated because it reflects the average of millions of examples rather than a deliberate, opinionated human choice. This skill systematically detects slop across three domains, scores severity, and provides balanced fix strategies with concrete before/after examples.

**Audience:** LLM coding agents (Claude Code, Cursor, Aider, etc.) reviewing or generating output.

**Domains covered:**
- Copywriting & marketing text
- Frontend UI, CSS, design patterns, and visual identity
- Backend code quality, architecture, and engineering patterns

**Output:** For each piece of content reviewed, produce a structured audit with severity scores, flagged patterns, and fix options with tradeoffs.

---

## How to Use This Skill

**Step 1 — Identify the domain(s)** in the content being reviewed: text, frontend, code, or a combination.

**Step 2 — Run detection** using the pattern checklists in each domain section. Flag every matched pattern.

**Step 3 — Score severity** using the 1–5 scale defined per domain. Sum the flags and map to an overall grade: Clean (0–2 flags), Mediocre (3–5), Sloppy (6–9), Slop (10+).

**Step 4 — Generate the audit report** using the format in the Audit Output Format section.

**Step 5 — Apply fixes** using the balanced strategies. Present 2–3 options with tradeoffs rather than a single prescription, unless the fix is unambiguous.

When reviewing code generated by an LLM agent mid-task, check each domain relevant to what was just generated and surface any severity-3+ flags before proceeding.

## Domain 1: Copywriting & Text AI Slop

### 1.1 Detection Patterns

Score **1 point per flag** unless marked with a severity multiplier.

**Vocabulary & Phrase Patterns**
- [ ] "Delve into" / "Let's delve" (×2 — notorious LLM tell)
- [ ] "Unlock the power of" / "Harness the power of"
- [ ] "In today's fast-paced world" / "In today's digital landscape"
- [ ] "It's important to note that" / "It's worth noting"
- [ ] "Crucial" / "pivotal" / "transformative" used without stakes
- [ ] "Vibrant" / "robust" / "comprehensive" as filler adjectives
- [ ] "Leverage" used when "use" would suffice
- [ ] "Streamline" / "optimize" / "enhance" as vague verbs
- [ ] "Cutting-edge" / "state-of-the-art" / "innovative solution"
- [ ] "Foster" used for "create" or "build"
- [ ] "Navigate" as a metaphor ("navigate the complexities of")
- [ ] "Landscape" as a metaphor ("the evolving landscape of")
- [ ] "Tapestry" / "interplay" / "nuanced" used generically
- [ ] "At the end of the day" / "When all is said and done"
- [ ] "Game-changer" / "paradigm shift" without evidence
- [ ] Vague citations: "Studies show" / "Experts agree" / "Research indicates" without sources (×2)
- [ ] "Dive deep" / "deep dive" as a structural metaphor
- [ ] Em dash overuse — particularly to manufacture—dramatic pauses (×2)

**Structural Patterns**
- [ ] Opens with a rhetorical question: "Have you ever wondered...?" / "Are you struggling with...?"
- [ ] Three-word punchy sentences used for false urgency ("Results. Fast. Guaranteed.")
- [ ] Bulleted list where flowing prose would serve better
- [ ] Excessive use of "not X, but Y" construction for false revelation
- [ ] Every paragraph roughly the same length (assembly-line rhythm)
- [ ] Conclusion restates intro almost verbatim
- [ ] "In conclusion" / "In summary" used as transitions (×2)
- [ ] Numbered list of 5–10 items that feels like a random selection
- [ ] Hedging on every claim: "may," "could," "might," "often," "sometimes" stacked together
- [ ] Excessive qualification with "However," "Furthermore," "Moreover," "Additionally" at paragraph starts

**Tone & Voice Patterns**
- [ ] Relentlessly positive — no tension, no tradeoffs, no real opinions
- [ ] Second-person overuse ("You will discover," "You'll be amazed") in all directions
- [ ] Formal when the brand is casual, or casual when the brand is formal (context mismatch)
- [ ] Zero humor, irony, or self-awareness
- [ ] No specific details, numbers, or named examples (pure abstraction)
- [ ] Personality-free — could have been written for any brand in any industry

### 1.2 Severity Scoring

| Score | Label | What it Means |
|-------|-------|---------------|
| 0–2 flags | ✅ Clean | Minor polish needed |
| 3–5 flags | 🟡 Mediocre | Clearly template-driven, rewrite passages |
| 6–9 flags | 🟠 Sloppy | Pervasive slop, structural rewrite needed |
| 10+ flags | 🔴 Slop | Full rewrite — this reads as unedited LLM output |

### 1.3 Before/After Examples

**BEFORE (Slop):**
> "In today's fast-paced digital landscape, it's crucial to leverage cutting-edge solutions that streamline your workflow and unlock the power of innovation. Our comprehensive platform fosters collaboration and drives transformative results."

**AFTER — Option A (Direct/Punchy):**
> "Most project tools slow you down. Ours doesn't. [Product] cuts handoff time by 40% with async workflows that actually work across time zones."

**AFTER — Option B (Narrative/Human):**
> "We built [Product] after our own team spent six months fighting Slack threads and missed deadlines. It's the tool we needed and couldn't find."

---

**BEFORE (Slop):**
> "Have you ever wondered how to navigate the complexities of modern team management? This guide will delve into the key strategies that experts agree are pivotal to success."

**AFTER:**
> "Running a distributed team is genuinely hard. Here's what actually worked for us after two years of trial and error — no frameworks, just honest lessons."

### 1.4 Fix Strategies

**Option A — Voice injection (minimal rewrite, high impact):**
Add one specific detail, one concrete number, and one moment of real opinion or tension. Replace 3 generic adjectives with either a specific word or nothing at all. Cost: low effort; Risk: may still feel slightly generic.

**Option B — Structural rewrite (medium effort):**
Replace the rhetorical opening with a specific scenario, claim, or fact. Cut all "In conclusion" transitions. Turn bulleted lists of abstract benefits into 1–2 concrete use cases with named results. Cost: medium; Risk: may lose SEO keyword density if that matters.

**Option C — Full brand voice pass (highest effort, best result):**
Write a 3-sentence "voice brief" for the brand (e.g., "skeptical, direct, slightly dry"), then rewrite the entire piece treating that brief as a hard constraint. Remove every generic adjective. Cost: high; Benefit: produces text that reads as distinctly human.

**Universal fixes regardless of option:**
- Replace "Studies show" with a real citation or remove the claim
- Replace all em dashes with commas, semicolons, or sentence breaks — then add back only where truly needed
- If the piece could be for any industry, it's not done yet

## Domain 2: Frontend UI & Design AI Slop

The root cause: LLMs are trained on Tailwind tutorials, GitHub repos, and SaaS landing page templates from 2019–2024. Without explicit design constraints, they output the statistical median — which is purple, Inter-fonted, and three-column.

### 2.1 Detection Patterns

**Color & Visual Identity**
- [ ] Primary color is indigo/violet/purple (`indigo-500`, `purple-600`, `#6366f1`, `#8b5cf6`) (×2)
- [ ] Gradient goes from purple-to-blue or pink-to-purple
- [ ] "Aurora" or mesh gradient background with no relationship to brand
- [ ] Glassmorphism (`backdrop-blur`, `bg-white/10`) applied as trend decoration rather than structural hierarchy
- [ ] Neumorphism (soft shadows, low contrast) used without accessibility consideration
- [ ] Color palette is "timid" — 3+ muted tones with no dominant color or sharp accent
- [ ] Background is solid white (`#ffffff`) or solid `gray-50` with no atmosphere
- [ ] No dark mode consideration when the context warrants it

**Typography**
- [ ] Font is Inter, Roboto, or Arial (the "safe defaults") (×2)
- [ ] No font pairing — single font family for everything
- [ ] All headings use `font-bold` with no contrast in weight variation
- [ ] Type scale is linear and mechanical (no expressive size jumps)
- [ ] Gradient text on every heading (`bg-clip-text text-transparent`)
- [ ] No consideration of line-height, letter-spacing, or measure (line length)

**Layout & Structure**
- [ ] Hero: centered text + CTA button + optional background graphic (×2)
- [ ] "Three features in three boxes" pattern below hero (×2)
- [ ] All sections are centered, full-width, with identical padding rhythm
- [ ] Cards everywhere — every piece of content wrapped in a rounded-corner card
- [ ] Bento grid used as pure aesthetic without content logic
- [ ] Icons from Heroicons, Lucide, or Font Awesome with no visual personality
- [ ] Sticky navbar with logo left + links center/right + CTA button (default SaaS template)
- [ ] Footer is a 4-column link dump with no brand personality
- [ ] Every corner is `rounded-xl` or `rounded-2xl` — nothing sharp or intentional
- [ ] Shadows are all `shadow-md` or `shadow-lg` at 0.1 opacity — timid everywhere

**Components & Interaction**
- [ ] No loading states, error states, or empty states on interactive components
- [ ] Form inputs have no validation indicators or error messaging
- [ ] Buttons have hover only — no focus, active, or disabled states
- [ ] Animations are all generic fade-ins (`opacity-0 → opacity-100 on scroll`)
- [ ] No keyboard navigation or ARIA labels on custom components
- [ ] Modals lack focus trapping or accessible close behavior
- [ ] Toast notifications use a generic library with default styling

**CSS & Code Patterns**
- [ ] Tailwind utility classes stacked 15+ deep without component extraction
- [ ] `!important` used to override Tailwind base styles
- [ ] Inline `style={}` used alongside Tailwind for one-off tweaks
- [ ] CSS variables not used — hardcoded color values scattered throughout
- [ ] No responsive breakpoints considered beyond `md:` and `lg:`
- [ ] Animation uses `transition-all` instead of specific properties

### 2.2 Severity Scoring

| Score | Label | What it Means |
|-------|-------|---------------|
| 0–3 flags | ✅ Clean | Solid, defensible choices |
| 4–7 flags | 🟡 Mediocre | Template-adjacent, needs intentional design decisions |
| 8–12 flags | 🟠 Sloppy | Clearly AI-default output, requires design pass |
| 13+ flags | 🔴 Slop | Indistinguishable from every other AI-generated SaaS page |

### 2.4 Fix Strategies

**Option A — Color system replacement (high impact, low code change):**
Define 3–5 CSS custom properties for brand colors. Choose a dominant color that is NOT in the indigo/violet family unless it's a deliberate brand decision. Use a tool like Coolors or Radix Colors. Pick one sharp accent that creates tension with the dominant. Risk: requires design direction; Benefit: solves 30% of slop flags at once.

**Option B — Typography upgrade (medium effort):**
Replace Inter/Roboto with one of: a geometric sans (DM Sans, Plus Jakarta Sans, Outfit), a humanist sans (Figtree, Nunito), a display serif for headings (Playfair Display, Lora), or a monospace for technical character (JetBrains Mono, Fira Code). Pair heading and body fonts with different weights. Add `letter-spacing` to headings and tune `line-height` for body. Risk: fonts can feel inconsistent without care; Benefit: biggest single visual signal of "a human designed this."

**Option C — Layout deconstruction (highest effort, most distinctive):**
Break the centered-hero + three-boxes pattern. Choose: left-aligned asymmetric hero, editorial text-first layout, split-screen, or scroll-driven reveal. Use CSS Grid with intentional column definitions rather than Flexbox centering. Risk: takes longer, requires design thinking; Benefit: immediately distinguishable from template output.

**Universal frontend fixes:**
- Add `:focus-visible` styles to all interactive elements
- Implement all interactive states (hover, focus, active, disabled, loading, error)
- Replace `transition-all` with specific transition properties
- Define all colors as CSS variables — never hardcode hex in components
- Remove `!important` — fix the specificity issue at the root instead

## Domain 3: Code Architecture & Quality AI Slop

The root cause: LLMs optimize for local correctness (does this snippet work?) rather than global coherence (does this fit the system?). They pattern-match against training data of wildly varying quality and lack architectural judgment about maintainability, security, and extensibility.

### 3.1 Detection Patterns

**Architecture & Structure**
- [ ] Functions over 50 lines that could be split into named sub-operations (×2)
- [ ] Logic duplicated across 2+ files because the AI didn't check existing implementations (×2)
- [ ] Monolithic component/class doing 3+ unrelated things (God Object)
- [ ] New abstraction introduced for a single use case (premature abstraction)
- [ ] Elaborate class hierarchy where a simple function or record type would suffice
- [ ] Feature added by appending to existing file rather than considering refactor (accumulation over design)
- [ ] No separation of concerns — business logic inside UI components or route handlers
- [ ] Mixed levels of abstraction in a single function (high-level orchestration + low-level string manipulation together)
- [ ] No consistent module/file organization pattern across the codebase

**Naming & Readability**
- [ ] Variable names: `data`, `result`, `temp`, `item`, `obj`, `val` used for anything non-trivial (×2)
- [ ] Boolean variables without `is/has/can/should` prefix
- [ ] Function names that describe implementation rather than intent (`processArray` vs `filterExpiredSessions`)
- [ ] Inconsistent naming conventions within a single file (camelCase + snake_case mixed)
- [ ] Magic numbers/strings hardcoded without named constants
- [ ] Overly verbose names that repeat type info: `userObject`, `listArray`, `stringValue`

**Error Handling & Edge Cases**
- [ ] `try/catch` that swallows errors silently (`catch (e) {}` or `catch (e) { console.log(e) }`) (×2)
- [ ] No error handling on async operations / unhandled promise rejections
- [ ] Happy path only — no handling for null, undefined, empty array, network failure
- [ ] Error messages that expose internal implementation details to end users
- [ ] Validation logic absent on user-facing inputs
- [ ] Hardcoded credentials, API keys, or secrets in source (×3 — critical security)

**Testing Quality**
- [ ] Tests that only cover the happy path — no edge cases, no error paths
- [ ] Test descriptions that restate the implementation ("it calls getUserById")
- [ ] Tests with no assertions (test always passes)
- [ ] Mocks that are more complex than the code being tested
- [ ] No integration tests — only unit tests that mock everything

**Dependency & Library Choices**
- [ ] Outdated library versions or deprecated APIs used (AI trained on old docs)
- [ ] Heavy library imported for a trivial utility (Lodash imported for one `_.get()`)
- [ ] Multiple libraries doing the same job (Axios + fetch, Moment + date-fns)
- [ ] `any` type used liberally in TypeScript (×2 — defeats the purpose)
- [ ] `// @ts-ignore` without explanation

**Performance & Security**
- [ ] Database query or API call inside a loop (N+1 pattern) (×2)
- [ ] No pagination on list queries that could return unbounded results
- [ ] User input directly interpolated into SQL/shell/HTML without sanitization (×3 — critical)
- [ ] Secrets or PII logged to console
- [ ] No rate limiting consideration on public endpoints
- [ ] Synchronous operations used where async is appropriate (blocking I/O)

**AI-Specific Tells**
- [ ] Empty class body that was never populated (`class EventEmitter {}`)
- [ ] Comment block that explains what the code does line-by-line rather than why
- [ ] `// TODO: implement error handling` left in committed code (×2)
- [ ] Inconsistent implementation of the same pattern in different files (context amnesia)
- [ ] Overly defensive code that checks for impossible conditions

### 3.2 Severity Scoring

| Score | Label | What it Means |
|-------|-------|---------------|
| 0–3 flags | ✅ Clean | Production-ready with minor polish |
| 4–6 flags | 🟡 Mediocre | Functional but incurring technical debt |
| 7–10 flags | 🟠 Sloppy | Needs architectural review before merge |
| 11+ flags | 🔴 Slop | Do not merge — rewrite with explicit architecture intent |
| Any ×3 flag | 🚨 CRITICAL | Security/data integrity issue — fix immediately |

### 3.3 Before/After Examples

**BEFORE (Slop) — God function with swallowed errors:**
```typescript
async function processData(data: any) {
  try {
    const result = await db.query("SELECT * FROM users WHERE id = " + data.id);
    const items = result.rows.map(item => {
      return { id: item.id, name: item.name, email: item.email };
    });
    return items;
  } catch (e) {
    console.log(e);
    return [];
  }
}
```

**AFTER — Option A (Minimal, pragmatic fix):**
```typescript
async function getUsersByIds(userIds: string[]): Promise<User[]> {
  if (!userIds.length) return [];

  const placeholders = userIds.map((_, i) => `$${i + 1}`).join(', ');
  const result = await db.query(
    `SELECT id, name, email FROM users WHERE id IN (${placeholders})`,
    userIds
  );

  return result.rows.map(mapRowToUser);
}

function mapRowToUser(row: UserRow): User {
  return { id: row.id, name: row.name, email: row.email };
}
```
*Note: caller handles errors — don't swallow them.*

---

**BEFORE (Slop) — Abstraction bloat:**
```typescript
class DataProcessorFactory {
  createProcessor(type: string): DataProcessor {
    switch(type) {
      case 'csv': return new CsvDataProcessor(new CsvParser(), new DataValidator());
      case 'json': return new JsonDataProcessor(new JsonParser(), new DataValidator());
    }
  }
}
```
*Used exactly once, for two cases.*

**AFTER — Option A (Function-first, YAGNI):**
```typescript
function parseUpload(file: File): ParsedData {
  if (file.type === 'text/csv') return parseCsv(file);
  if (file.type === 'application/json') return parseJson(file);
  throw new Error(`Unsupported file type: ${file.type}`);
}
```
*Extract the factory only when you have 4+ cases or the instantiation becomes complex.*

### 3.4 Fix Strategies

**Option A — Naming and readability pass (quick win, high ROI):**
Rename all `data/result/temp/item` variables to domain-specific names. Add `is/has/can` to booleans. Replace magic numbers with named constants. This alone reduces cognitive load significantly. Cost: 30–60 min for a medium file; Risk: minimal, purely additive.

**Option B — Error handling hardening (medium effort, high impact):**
Audit every `try/catch` and `async` call. Ensure errors are logged with context (not swallowed), propagated to callers appropriately, and exposed to users as human-readable messages (not stack traces). Add input validation at every system boundary. Cost: medium; Risk: may surface latent bugs (this is good).

**Option C — Structural refactor (highest effort, required for long-term health):**
Identify the single most-violated separation of concern (usually: UI doing data fetching, or route handlers doing business logic). Extract it. Follow with a second extraction. Don't try to fix all architecture at once — incremental refactoring is more reliable. Cost: high; Risk: requires test coverage before refactoring safely.

**For the ×3 security flags — these are non-negotiable:**
- SQL injection: Always use parameterized queries — never string interpolation
- Hardcoded secrets: Move to environment variables immediately, rotate the exposed secret
- Unsanitized HTML: Use a trusted sanitization library (DOMPurify) or framework escaping

**When to use each option:**
- Prototype/throwaway code: Option A only
- Feature in active development: Options A + B
- Production system with users: All three, prioritized by severity score

When producing an AI slop audit, always use this structure:

```
## AI Slop Audit

### Overall Grade: [✅ Clean / 🟡 Mediocre / 🟠 Sloppy / 🔴 Slop]
**Total flags:** X  |  **Domains:** Text (X) · Frontend (X) · Code (X)
[🚨 CRITICAL ISSUES: N] ← only show if any ×3 flags triggered

---

### 🚨 Critical Issues (Fix Immediately)
[List any ×3 severity flags with direct fix instructions — no options, just the fix]

---

### 🔴/🟠 High-Priority Flags
[List flags with severity ≥ 2, grouped by domain]

### 🟡 Medium-Priority Flags
[List remaining flags]

---

### Recommended Fixes

**Quick Wins (< 1 hour)**
[2–3 specific, actionable fixes with no architectural risk]

**Strategic Fixes (choose one)**
Option A: [Name] — [what it does] | Tradeoff: [cost vs. benefit]
Option B: [Name] — [what it does] | Tradeoff: [cost vs. benefit]
Option C: [Name] — [what it does] | Tradeoff: [cost vs. benefit]

---

### Before/After
[Show 1–2 specific rewrites from the content being reviewed, not generic examples]
```

**Rules for producing the audit:**
- Always show specific examples from the content reviewed, not generic templates
- Never recommend "just rewrite everything" without a staged plan
- For LLM-generated frontend code, always check colors first (highest slop density)
- For LLM-generated copy, always check vocabulary patterns first
- For LLM-generated backend code, check for ×3 security flags before anything else
- If asked to fix rather than just audit, apply all Quick Wins automatically and present Strategic Fix options

## Quick Reference Cheatsheet

### Instant Slop Tells by Domain

**Copy:** "delve", "leverage", "transformative", "in today's landscape", em dash overuse, vague citations, rhetorical question opener

**Frontend:** `indigo-500` / purple gradient, Inter/Roboto font, centered hero + three boxes, `backdrop-blur` glassmorphism, `rounded-xl` everything, `shadow-md` everywhere, `transition-all`

**Code:** `data/result/temp` variable names, swallowed `catch (e) {}`, `any` TypeScript type, SQL string interpolation, hardcoded secrets, God functions 50+ lines, duplicated logic across files, `// TODO: add error handling`

### The "Would a Junior Dev Do This?" Test
AI code anti-patterns mirror what a fast, uncritical junior developer would produce under time pressure: it works, it compiles, it passes the happy path test — but it doesn't consider edge cases, security, or the next person who has to modify it. Apply this frame when uncertain if something is slop.

### The "Could This Be Any Brand?" Test
AI copy anti-patterns mirror what a content-farm writer would produce for a generic SaaS company. If you could swap out the product name and the copy still makes sense for a competitor, it's slop. Good copy is specific, opinionated, and would be wrong if applied to a different brand.

### The "Statistical Median" Test
AI frontend anti-patterns are the visual median of every Tailwind tutorial and SaaS landing page template from 2019–2024. Ask: "Is this choice deliberate, or is it just the default?" Defaults are slop. Deliberate choices — even conventional ones — are not.
